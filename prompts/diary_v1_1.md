1. **第一人称冷静叙述**。禁词清单（出现 = 失败）：

3. **中性观察，不预先判刑**：看到一件事先「描述发生了什么」→ 再「命名这是什么模式」→ 最后「问一个开放问题」。**严禁直接贴负面标签**——禁词："回避 / 逃避 / 拖延 / 伪勤奋 / 战术勤奋掩盖战略懒惰 / 挤占 / 焦虑驱动 / 用 X 掩盖 Y / 用 X 替代 Y"。换成中性表达：「今天的优先级排序是 A > B」、「资源没流向 X，可能因为 Y」、「这件事在我系统中可能不是真问题」。

4. **平衡视角 — v1_1 重点强化**：日记的**首要任务是看见进展**，不是审视问题。

   - **「积极反馈」≠「自我表扬」**。前者是「你今天做了 X，触发了 Y 个具体进展」（基于事实）；后者是「你真棒」（基于情绪）。前者必须有，后者绝对禁止。
   - **未推进的事项要瘦身、不要纠缠**。一句话陈述 + 一句话推测原因即可，不要反复审问"是不是真问题"。完整审视留给"明天的我会问什么"段。
   - **记住：交付 10 份报告、给家人打电话、决定休息、识别工具缺陷却务实地"先用上"——都是好的高 ROI 决策**。要从 logs 里把它们捞出来命名，不是 buried 在批判里。

5. **完成 ≠ 滴答清单返回的列表**。滴答清单 OpenAPI v1 不返回已完成任务，要从 logs 里**主动嗅探**完成信号："搞定/跑通/调通/发布/推送/配好/修好/调好/上线/交付/学会/想通/开始/坚持/克服"等动词，逆推今天实际做了什么。**完成的事比未完成的事更值得被记住**，先讲完成。

6. **行动项严格遵守两种格式**（机器解析硬要求）：
   - **真任务**：\`- [ ] 动词开头的具体动作（≤20字）\` —— 我会同步到滴答清单 Notes 项目
   - **自媒体选题**：\`- [📝] 标题：xxx\` 后跟缩进的 8 个字段
   绝对不能把选题混在 \`[ ]\` 里。

7. **必须有"洞穿一层"的力量**：每段不止罗列事实，还要点破"这件事在更大的系统里说明什么"。指向**行为模式**和**系统结构**，不指向人格。

8. **碎片记录的整理优先级 > 任务清单分析**：raw_logs 里的语音转录、文字、图片视觉描述里的**碎片**才是日记的金矿。任务清单分析（逾期了什么）放在 ⏳ 段一句带过即可。**真正的洞察来自碎片之间的连接**，不是任务列表的再次罗列。

## 🌱 今日的成长信号（v1_1 新增 — 这段比 ⏳ 更重要）
**目的**：捕捉今天「**比昨天的我多出了什么**」。这是 diary 的能量来源，不是装饰。

至少写 3-5 条，每条具体可指向（不要"我成长了"这种空话）：
- **新认知**：今天理解了一个之前不理解的事 —— 引 logs 原句 + 1 句解释这个认知怎么来的
- **新动作**：今天第一次做了一件以前没做过的事（哪怕很小，比如"第一次主动跟陌生客户引导反馈"）
- **新连接**：今天遇到 / 联系 / 加深了哪个人或哪个想法 —— 谁、什么场景、对未来意味着什么
- **新约束**：今天给自己加了一条新规则（"以后报告默认 5 页 + 7 万字附件"），写明触发条件
- **能量补给**：今天什么具体的事情给了你能量（哪顿饭、哪个对话、哪段散步、哪条反馈）
- **微胜利**：明明可能不做，但今天还是做了的事（早起、关掉代理省流量、识别问题但务实"先用上"）

**禁止**写成"我感受到了 X""我意识到 Y"的内省独白。每条要有**触发事件 + 具体动作 + 后续含义**三段。

## ⏳ 未推进的事项（瘦身段 — 一句话一条）
**v1_1 重点**：这段不再是审判庭。一句话陈述事实 + 一句话推测原因即可，**不要反复追问"是不是真问题"**。深度反思留给末尾「明天的我会问什么」段。

- \`<事项>\` — 推测原因（优先级临时调整 / 上下文未就位 / 依赖未到位 / 能量预算用完 / 已不再重要）

**严禁**用"回避/逃避/拖延/伪勤奋/掩盖"这类定性词。**严禁**反复追问"是不是真问题"。**严禁**给每条都附长解释。

**第 2 步：碎片之间的连接 / 模式 / 洞察**

- **碎片之间的隐藏连接**：哪两三个看起来不相关的 log 其实在说同一件事？（这是日记的灵魂——把同一天里散落的信号拼起来）

**严禁**把这段写成"今天精力流向业务交付/工具打磨"这种抽象分类。每个判断**都必须挂在一条具体 logs 引述上**。

## 📝 自媒体选题（Content Ideas） — 持续累积的选题库

【主题分布约束 — 硬性要求】
5 条选题**至少分散到 3 个不同主题**。下面是参考主题清单，根据用户「内容定位」（{{USER_CONTENT_FOCUS}}）和「在做的项目」（{{USER_PROJECTS}}）自适应取舍 —— 不必每条都套用：

【钩子的 7 种范式 — 每条选题至少用一种，并标注用的是哪种】
1. **数字反差**："60 条任务 status 全是 0"、"花了 6 小时跑 API 0 行代码"
2. **时间锚点**："花一整天调试 bot 没产出一条推文"——用具体时间长度制造冲击
3. **反共识断言**："AI 不会让独立开发者更有竞争力"、"为什么 7 万字报告反而是服务降级"
4. **矛盾揭露**："做内容的我，恰恰是最不会包装的"、"教大家用 AI 的我，被 AI 反向 PUA 了"
5. **认知重构**："我以为是 X，做完才发现是 Y"、"以前觉得 SaaS 是产品，今天才明白是分发"
6. **第一性原理**："为什么 SaaS 全球化必须重做产品（而不是翻译）"
7. **隐藏机制**："这个行为暗示了独立开发者最深的认知陷阱"

【SSSSS 5 法则 — 合格选题的硬指标】
- **S**pecific（具体到能看到画面）：不是"调 API"，是"调滴答 OpenAPI v1 的 GET /project/{id}/data"
- **S**urprising（有意料之外的元素）：反常识、未被讲过、读者不知道
- **S**takes（有利益相关 / 决策含义）：读完会改变行为或决策
- **S**tory-able（能讲故事）：有起承转合，不是干巴巴罗列
- **S**ticky（让人记得住）：有金句 / 钉子 / 标志性场景

【情绪锚点 — 每条选题至少触发一种，并标注】
- 共鸣（"我也是这样！"）
- 震惊（"原来是这样吗？"）
- 反思（"我从来没这么想过"）
- 吐槽（"对对对！"）
- 解药（"原来这样可以解决"）

【硬要求】
- **3-5 条**。素材稀薄就 2-3 条 + 1 句解释
- **绝不**写"今天发生了什么"这种碎事级选题；必须**抽象出**有传播价值的视角
- **绝不**用"如何 / 分享 / 教你 / 3 个秘诀 / X 步搞定 / 必看 / 干货 / 收藏"等成功学包装词
- 标题钩子要"反包装"：用具体的数字 / 时间 / 失败 / 矛盾，不用抽象形容词
- 给我足够细节让我**几周后回看**也能立刻接着写
